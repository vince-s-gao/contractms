## 1. 数据库设计原则

### 1.1 命名规范
- **表名**：小写，下划线分隔，使用复数形式（如：users、contracts、payment_plans）
- **字段名**：小写，下划线分隔
- **索引名**：idx_表名_字段名
- **外键名**：fk_表名_字段名
- **唯一键名**：uk_表名_字段名

### 1.2 通用字段
所有表都包含以下通用字段：
- `id`：主键，BIGINT，自增
- `tenant_id`：租户ID，BIGINT COMMENT '租户ID（用于SaaS多租户数据隔离）'
- `created_at`：创建时间，DATETIME
- `updated_at`：更新时间，DATETIME
- `created_by`：创建人ID，BIGINT
- `updated_by`：更新人ID，BIGINT

### 1.3 状态字段规范
- 使用 TINYINT 类型
- 1 表示启用/有效
- 0 表示禁用/无效
- 使用 COMMENT 注释说明状态含义

## 2. 用户和权限模块

### 2.1 用户表（users）
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(50) UNIQUE NOT NULL COMMENT '用户名',
    password VARCHAR(255) NOT NULL COMMENT '密码（加密存储）',
    real_name VARCHAR(50) NOT NULL COMMENT '真实姓名',
    email VARCHAR(100) COMMENT '邮箱',
    phone VARCHAR(20) COMMENT '手机号',
    department VARCHAR(100) COMMENT '部门',
    status TINYINT DEFAULT 1 COMMENT '状态：1-启用，0-禁用',
    last_login_at DATETIME COMMENT '最后登录时间',
    last_login_ip VARCHAR(50) COMMENT '最后登录IP',
    deleted_flag TINYINT DEFAULT 0 COMMENT '逻辑删除标记：0-正常，1-删除',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    created_by BIGINT COMMENT '创建人ID',
    updated_by BIGINT COMMENT '更新人ID',
    INDEX idx_username (username),
    INDEX idx_department (department),
    INDEX idx_status (status),
    INDEX idx_deleted_flag (deleted_flag)
) COMMENT '用户表';
```

## 4. 合同管理模块

### 4.1 合同表（contracts）
```sql
CREATE TABLE contracts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '合同ID',
    contract_no VARCHAR(50) UNIQUE NOT NULL COMMENT '合同编号',
    contract_name VARCHAR(200) NOT NULL COMMENT '合同名称',
    contract_type_id BIGINT NOT NULL COMMENT '合同类型ID',
    amount DECIMAL(18,2) NOT NULL COMMENT '合同金额',
    currency VARCHAR(10) DEFAULT 'CNY' COMMENT '币种',
    sign_date DATE COMMENT '签约日期',
    effective_date DATE COMMENT '生效日期',
    expiry_date DATE COMMENT '到期日期',
    party_name VARCHAR(200) NOT NULL COMMENT '对方单位名称',
    party_contact VARCHAR(100) COMMENT '对方联系人',
    party_phone VARCHAR(20) COMMENT '对方联系方式',
    our_signatory VARCHAR(100) COMMENT '我方签约人',
    contract_summary TEXT COMMENT '合同摘要',
    contract_terms TEXT COMMENT '合同条款',
    version_no INT DEFAULT 1 COMMENT '合同版本号',
    legal_terms TEXT COMMENT '法律条款',
    status VARCHAR(20) DEFAULT 'DRAFT' COMMENT '合同状态：DRAFT-草稿，SUBMITTED-已提交审批，APPROVED-审批通过，REJECTED-审批驳回，VOIDED-作废，SIGNED-已签署，EFFECTIVE-已生效，ACTIVE-执行中，TERMINATED-提前终止，COMPLETED-正常完成，ARCHIVED-归档',
    owner_id BIGINT COMMENT '负责人ID',
    department_id BIGINT COMMENT '部门ID',
    parent_contract_id BIGINT COMMENT '父合同ID（用于框架协议、子合同、续签合同、补充协议等关系）',
    relation_type VARCHAR(20) COMMENT '关系类型：FRAMEWORK-框架协议，SUB_CONTRACT-子合同，RENEWAL-续签合同，SUPPLEMENT-补充协议',
    deleted_flag TINYINT DEFAULT 0 COMMENT '逻辑删除标记：0-正常，1-删除',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    created_by BIGINT COMMENT '创建人ID',
    updated_by BIGINT COMMENT '更新人ID',
    INDEX idx_contract_no (contract_no),
    INDEX idx_contract_type_id (contract_type_id),
    INDEX idx_status (status),
    INDEX idx_owner_id (owner_id),
    INDEX idx_party_name (party_name),
    INDEX idx_sign_date (sign_date),
    INDEX idx_expiry_date (expiry_date),
    INDEX idx_parent_contract_id (parent_contract_id),
    INDEX idx_relation_type (relation_type),
    INDEX idx_deleted_flag (deleted_flag),
    FOREIGN KEY (contract_type_id) REFERENCES contract_types(id),
    FOREIGN KEY (owner_id) REFERENCES users(id),
    CONSTRAINT fk_contract_parent FOREIGN KEY (parent_contract_id) REFERENCES contracts(id) ON DELETE SET NULL
) COMMENT '合同表';
```

## 14. SaaS多租户架构设计

### 14.1 多租户设计概述

为支持系统未来SaaS化改造，采用基于租户ID的数据隔离方案。所有业务表都增加`tenant_id`字段，通过租户ID实现数据层面的逻辑隔离。

#### 14.1.1 设计原则
- **数据隔离**：每个租户只能访问自己的数据
- **扩展性**：支持大规模租户部署
- **性能**：通过租户ID索引保证查询性能
- **安全性**：防止跨租户数据泄露

### 14.2 租户数据隔离机制

#### 14.2.1 数据隔离策略
- **物理隔离**：每个租户使用独立的数据库（适用于大型企业客户）
- **逻辑隔离**：所有租户共享同一数据库，通过`tenant_id`字段区分（适用于中小型客户）
- **混合模式**：根据租户规模和需求选择不同的隔离策略

#### 14.2.2 租户ID应用规则
- 所有业务表必须包含`tenant_id`字段
- 所有查询操作必须包含租户ID过滤条件
- 租户ID在系统层面自动注入，避免手动传递
- 租户ID不可为空，确保数据隔离完整性

### 14.3 多租户查询和权限控制

#### 14.3.1 查询过滤机制
```sql
-- 所有业务查询必须包含租户ID过滤
SELECT * FROM contracts WHERE tenant_id = ? AND status = 'ACTIVE';

-- 联合查询时需要在每个表上添加租户ID过滤
SELECT c.*, p.payment_amount 
FROM contracts c 
LEFT JOIN payment_plans p ON c.id = p.contract_id 
WHERE c.tenant_id = ? AND p.tenant_id = ?;
```

#### 14.3.2 权限控制策略
- **租户管理员**：拥有租户内所有数据的完全访问权限
- **租户用户**：根据角色权限访问租户内数据
- **系统管理员**：可跨租户访问数据（需特殊权限控制）

### 14.5 多租户架构实施建议

#### 14.5.1 技术实现方案
- **租户上下文管理**：使用ThreadLocal或请求上下文管理租户信息
- **数据源路由**：基于租户ID动态选择数据源
- **查询拦截器**：自动为所有查询添加租户ID过滤条件
- **缓存策略**：按租户隔离缓存，避免数据混淆

#### 14.5.2 性能优化建议
- **租户ID索引**：为所有业务表的`tenant_id`字段创建索引
- **分库分表**：当租户数量庞大时，考虑按租户分库分表
- **读写分离**：为大型租户提供读写分离支持
- **连接池优化**：按租户粒度管理数据库连接

#### 14.5.3 安全控制措施
- **租户边界检查**：所有API调用必须验证租户权限
- **数据泄露防护**：防止SQL注入导致的跨租户数据访问
- **审计日志**：记录所有跨租户操作日志
- **权限验证**：确保用户只能访问所属租户的数据

### 14.6 迁移和升级策略

#### 14.6.1 单租户到多租户迁移
1. **数据迁移**：为现有数据分配默认租户ID
2. **代码改造**：在所有数据访问层添加租户ID过滤
3. **权限调整**：重构用户权限体系支持多租户
4. **测试验证**：确保迁移后数据隔离正常

#### 14.6.2 版本升级策略
- **向后兼容**：确保新版本兼容单租户模式
- **渐进式升级**：支持分批升级，降低风险
- **回滚机制**：提供完善的版本回滚方案
- **数据备份**：升级前必须进行完整数据备份
