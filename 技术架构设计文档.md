## 1. 架构概述

### 1.1 架构设计原则

本系统采用现代化的企业级技术架构，确保系统的高可用性、可扩展性和安全性。技术架构设计遵循以下核心原则：

- **高可用性**：通过集群部署、负载均衡、故障转移等机制确保系统稳定运行
- **可扩展性**：MVP阶段采用单体架构，企业版支持拆分为微服务，支持水平扩展，满足业务增长需求
- **安全性**：多层次安全防护，确保数据和系统安全
- **高性能**：通过缓存、索引优化、异步处理等技术提升系统性能
- **可维护性**：模块化设计，便于维护和升级
- **标准化**：遵循行业标准和最佳实践

### 1.2 架构阶段说明

系统架构采用分阶段演进模式：

**MVP阶段（第一阶段）**
- 采用单体架构（Monolith）
- 单一应用服务部署
- 降低系统复杂度
- 快速迭代产品能力

**企业版阶段（第二阶段）**
随着系统规模扩大，可拆分为微服务架构：
- 合同服务
- 用户服务
- 审批服务
- 通知服务
- 文件服务

### 1.3 系统逻辑架构

```text
┌───────────────────────────────────────┐

│ Web / 移动端 / 第三方系统 │

## 2. 前后端分离架构

### 2.1. 架构说明

系统采用前后端分离的架构模式，前端和后端通过RESTful API进行通信，实现解耦和独立部署。

### 2.2. 前端技术栈

- **框架**：Vue 3.x + Element Plus
- **状态管理**：Pinia
- **路由管理**：Vue Router
- **UI组件库**：Element Plus
- **构建工具**：Vite
- **HTTP客户端**：Axios
- **表单验证**：VeeValidate

### 2.3. 后端技术栈

- **框架**：Spring Boot 3.x
- **ORM框架**：MyBatis-Plus
- **API文档**：Swagger
- **安全框架**：Spring Security
- **缓存**：Redis
- **消息队列**：RabbitMQ

> 企业版可根据客户技术栈适配

### 2.4. 通信协议

- **API协议**：RESTful API
- **数据格式**：JSON
- **认证方式**：JWT (JSON Web Token)
- **跨域处理**：CORS

### 2.5. 架构优势

- **独立开发**：前后端团队可以并行开发，提高开发效率
- **灵活部署**：前端可以部署到CDN，后端可以独立扩展
- **技术选型灵活**：前后端可以根据需求选择最适合的技术栈
- **用户体验优化**：前端可以实现单页应用（SPA），提供更好的用户体验

## 3. 分布式部署架构

### 3.1. 架构说明

系统支持分布式部署，通过负载均衡、服务集群、数据库读写分离等技术手段，实现系统的高可用和高性能。

> **说明**：以下分布式架构为企业级增强能力，MVP阶段不涉及。MVP阶段采用单体架构部署即可满足需求。

### 3.2. 部署架构图

```
                    ┌─────────────┐
                    │   负载均衡   │
                    │  (Nginx)    │
                    └──────┬──────┘
                           │
             ┌──────────────┼──────────────┐
             │              │              │
       ┌─────▼─────┐  ┌─────▼─────┐  ┌─────▼─────┐
       │  应用节点1  │  │  应用节点2  │  │  应用节点3  │
       │ (后端服务) │  │ (后端服务) │  │ (后端服务) │
       └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
             │              │              │
             └──────────────┼──────────────┘
                            │
             ┌──────────────┼──────────────┐
             │              │              │
       ┌─────▼─────┐  ┌─────▼─────┐  ┌─────▼─────┐
       │  主数据库  │  │  从数据库  │  │   Redis   │
       │ (Master)   │  │  (Slave)   │  │   缓存    │
       └───────────┘  └───────────┘  └───────────┘
```

### 3.3. 负载均衡

- **负载均衡器**：Nginx
- **负载均衡策略**：轮询、最少连接、IP哈希
- **健康检查**：定期检查后端服务健康状态
- **会话保持**：通过Redis实现分布式会话

### 3.4. 应用集群

- **水平扩展**：支持动态增加应用节点
- **无状态设计**：应用节点无状态，便于扩展
- **服务发现**：企业版支持使用Consul实现服务注册与发现
- **配置中心**：企业版支持使用Nacos实现统一配置管理

### 3.5. 数据库集群

- **读写分离**：主库负责写操作，从库负责读操作
- **数据同步**：通过MySQL主从复制实现数据同步
- **连接池**：使用Druid管理数据库连接
- **分库分表**：大规模场景可按需设计分库分表方案

### 3.6. 缓存集群

- **缓存策略**：多级缓存（本地缓存 + Redis）
- **缓存穿透**：使用布隆过滤器
- **缓存雪崩**：设置随机过期时间
- **缓存击穿**：使用互斥锁

### 3.7. 分布式优势

- **高可用性**：单点故障不影响整体服务
- **高性能**：通过集群扩展提升系统吞吐量
- **可扩展性**：根据业务需求动态扩展资源
- **容错能力**：自动故障转移和恢复

## 4. 附件对象存储

### 4.1. 存储方案

系统采用对象存储方案管理合同附件，支持大文件上传、断点续传、文件预览等功能。

### 4.2. 存储技术选型

- **SaaS模式**：使用阿里云OSS（可替代：腾讯云COS、AWS S3）
- **私有化部署**：使用MinIO（可替代：Ceph）
- **混合方案**：支持多云存储和本地存储

### 4.3. 存储架构

```
┌─────────────┐
│   前端应用   │
└──────┬──────┘
       │
       │ 上传请求
       ▼
┌─────────────┐
│   后端端服务   │
└──────┬──────┘
       │
       │ 生成上传URL
       ▼
┌─────────────┐
│  对象存储    │
│  (OSS/COS)  │

### 4.4. 文件上传策略

- **直传模式**：前端直接上传到对象存储，减轻服务器压力
- **分片上传**：支持大文件分片上传和断点续传
- **上传限制**：单文件最大100MB（可配置），总存储空间可配置

- **访问控制**：通过签名URL实现临时访问权限
- **加密存储**：敏感文件支持加密存储
- **病毒扫描**：上传文件进行病毒扫描（可选）
- **水印处理**：支持添加水印保护文件

### 4.6. 文件管理

- **版本管理**：支持文件版本控制
- **生命周期**：自动清理过期文件
- **CDN加速**：通过CDN加速文件访问
- **预览服务**：集成在线预览服务

### 4.7. 存储优势

- **高可靠性**：高可靠对象存储，支持多副本冗余
- **高扩展性**：存储空间可无限扩展
- **高性能**：支持高并发上传和下载
- **成本优化**：按使用量计费，降低存储成本

## 5. RBAC权限模型

### 5.1. 模型概述

系统采用基于角色的访问控制（RBAC）模型，通过用户、角色、权限三层结构实现灵活的权限管理。

### 5.2. RBAC模型结构

```
组织(Organization)
│
▼
用户(User)
│
▼
角色(Role)
│
▼
权限(Permission) 数据权限(DataScope)
```

### 5.3. 权限类型

- **功能权限**：控制用户可以访问的功能模块
  - 菜单权限：控制菜单显示
  - 按钮权限：控制操作按钮显示
  - 接口权限：控制API访问

- **数据权限**：控制用户可以访问的数据范围
  - 全部数据：可以访问所有数据
  - 部门数据：只能访问本部门数据
  - 个人数据：只能访问个人创建的数据
  - 自定义数据：根据规则自定义数据范围

### 5.4. 权限验证流程

```
用户请求
    │
    ▼
获取用户信息
    │
    ▼
获取用户角色
    │
    ▼
获取角色权限
    │
    ▼
验证功能权限
    │
    ▼
验证数据权限
    │
    ▼
允许/拒绝访问
```

### 5.5. 权限缓存

- **缓存策略**：使用Redis缓存用户权限信息
- **缓存更新**：权限变更时自动更新缓存
- **缓存过期**：设置合理的缓存过期时间
- **缓存预热**：系统启动时预加载常用权限

### 5.6. 权限管理功能

- **角色管理**：创建、编辑、删除角色
- **权限分配**：为角色分配功能权限和数据权限
- **用户授权**：为用户分配角色
- **权限查询**：查询用户权限和角色权限

### 5.7. RBAC优势

- **灵活性**：通过角色和权限的组合实现灵活的权限控制
- **可维护性**：权限变更只需调整角色配置
- **可扩展性**：支持动态添加新的角色和权限
- **安全性**：细粒度的权限控制确保系统安全

## 6. 审计日志独立存储

系统将审计日志独立存储，与业务数据库分离，确保日志数据的完整性和可追溯性。

**核心特性**：
- **独立存储**：日志数据存储在独立数据库，与业务数据分离，互不影响
- **分区存储**：按时间分区存储，便于查询和管理
- **在线保留**：保留最近90天的日志在线存储，支持快速查询；超过90天的日志归档到冷存储

**日志类型**：
- **操作日志**：登录日志、操作日志、审批日志
- **系统日志**：错误日志、性能日志、安全日志

**查询功能**：支持条件查询、全文检索、统计分析、日志导出

## 9. 安全设计

### 9.1. 认证机制

- **JWT认证**：使用JSON Web Token进行用户认证
- **Token刷新**：支持Token自动刷新机制
- **多端登录**：支持同一用户多端同时登录
- **登录限制**：支持登录失败次数限制

**Token生命周期策略**
- Access Token 有效期：2小时
- Refresh Token 有效期：7天
- Token过期后需通过Refresh Token重新获取
- 用户登出或管理员禁用用户时立即失效

### 9.2. 授权机制

- **RBAC授权模型**：基于角色的访问控制
- **权限验证**：接口级别的权限验证
- **数据权限**：支持数据级别的权限控制
- **权限缓存**：使用Redis缓存权限信息

### 9.3. 数据加密

- **密码加密**：使用BCrypt加密用户密码
- **敏感数据加密**：敏感字段使用AES加密存储
- **传输加密**：使用HTTPS加密数据传输
- **数据脱敏**：敏感数据展示时进行脱敏处理

### 9.4. 防护措施

- **SQL注入防护**：使用参数化查询防止SQL注入
- **XSS防护**：对用户输入进行XSS过滤
- **CSRF防护**：使用Token防止CSRF攻击
- **接口限流**：使用Redis实现接口限流
- **防重放攻击**：使用时间戳和随机数防止重放攻击

### 9.5. 审计日志

- **操作审计**：记录所有用户操作
- **登录审计**：记录用户登录信息
- **异常审计**：记录系统异常信息
- **日志保护**：审计日志独立存储，防止篡改

## 10. 性能优化

系统通过多层次的性能优化策略，确保在高并发场景下的稳定响应：

- **缓存优化**：采用多级缓存架构（本地缓存 + Redis），支持缓存预热、穿透/雪崩/击穿防护
- **数据库优化**：索引优化、查询优化、连接池管理、读写分离
- **异步处理**：消息队列处理异步任务，日志异步写入，通知异步发送
- **前端优化**：路由懒加载、代码分割、CDN加速、浏览器缓存

> 性能指标：页面加载时间 < 2秒，接口响应时间 < 500ms，支持1000+并发用户

## 12. 实施周期

### 12.1. 实施阶段规划

系统实施分为五个阶段，确保项目按时、高质量交付：

| 阶段 | 周期 | 主要工作 | 交付物 |
|------|------|----------|--------|
| **项目启动** | 1周 | 需求确认、项目组建、计划制定 | 项目计划书、需求确认书 |
| **环境准备** | 1-2周 | 服务器部署、网络配置、安全设置 | 部署环境、系统访问地址 |
| **系统配置** | 2-3周 | 组织架构配置、角色权限配置、业务流程配置 | 配置文档、测试报告 |
| **数据迁移** | 1-2周 | 历史数据清洗、迁移、验证 | 数据迁移报告、数据对照表 |
| **培训上线** | 1-2周 | 用户培训、试运行、正式上线 | 培训记录、上线报告 |

**总实施周期**：6-10周（根据企业规模和数据量调整）

## 14. 数据迁移方案

### 14.1. 迁移流程

数据迁移采用标准化流程，确保数据完整性和准确性：

```
数据调研 → 方案设计 → 数据清洗 → 迁移执行 → 数据验证 → 上线切换
```

| 阶段 | 主要工作 | 输出成果 |
|------|----------|----------|
| **数据调研** | 源系统分析、数据质量评估、迁移范围确定 | 数据调研报告 |
| **方案设计** | 字段映射、转换规则、迁移策略制定 | 迁移方案文档 |
| **数据清洗** | 数据去重、格式标准化、异常数据处理 | 清洗后数据集 |
| **迁移执行** | 分批迁移、进度监控、异常处理 | 迁移日志 |
| **数据验证** | 数据完整性检查、业务验证、用户确认 | 验证报告 |
| **上线切换** | 最终同步、切换准备、回滚预案 | 上线检查清单 |

### 14.2. 支持的迁移数据类型

| 数据类型 | 迁移支持 | 说明 |
|----------|----------|------|
| 合同主数据 | ✓ | 合同基本信息、合同条款 |
| 合同附件 | ✓ | 历史合同文件、扫描件 |
| 付款记录 | ✓ | 历史付款计划及执行记录 |
| 供应商/客户 | ✓ | 合作伙伴基础信息 |
| 组织架构 | ✓ | 部门、人员基础信息 |
| 审批记录 | 部分 | 历史审批流程记录（视源系统支持情况） |

### 14.3. 数据安全保障

- **迁移前备份**：源系统和目标系统完整备份
- **迁移过程加密**：敏感数据传输加密
- **数据脱敏**：测试环境数据脱敏处理
- **回滚机制**：支持迁移失败快速回滚
- **审计追踪**：完整的迁移操作日志

### 14.4. 迁移时间估算

| 数据规模 | 预估迁移时间 |
|----------|--------------|
| 1万份合同以内 | 3-5个工作日 |
| 1-5万份合同 | 5-10个工作日 |
| 5万份以上 | 10-15个工作日 |

> 实际迁移时间需根据数据质量和复杂度评估
