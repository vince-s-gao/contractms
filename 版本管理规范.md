# 版本管理规范

## 1. 版本号规范

本项目遵循语义化版本规范（Semantic Versioning 2.0.0）：

- **主版本号 (MAJOR)**：不兼容的 API 修改
- **次版本号 (MINOR)**：向下兼容的功能性新增
- **修订号 (PATCH)**：向下兼容的问题修正

格式：`MAJOR.MINOR.PATCH`

## 2. 版本发布流程

### 2.1 开发阶段
- 在 `develop` 分支进行功能开发
- 功能完成后合并到 `main` 分支

### 2.2 版本发布
1. 更新 `VERSION` 文件中的版本号
2. 在 `CHANGELOG.md` 中添加版本变更记录
3. 创建版本标签：`git tag vX.Y.Z`
4. 推送标签：`git push origin vX.Y.Z`

## 3. 分支管理策略

### 3.1 主要分支
- `main`：主分支，存放稳定的发布版本
- `develop`：开发分支，集成功能开发

### 3.2 辅助分支
- `feature/*`：功能开发分支
- `hotfix/*`：紧急修复分支
- `release/*`：版本发布准备分支

## 4. 提交信息规范

提交信息格式：`类型(范围): 描述`

### 4.1 类型说明
- `feat`：新功能
- `fix`：修复bug
- `docs`：文档更新
- `style`：代码格式调整
- `refactor`：重构代码
- `test`：测试相关
- `chore`：构建过程或辅助工具变动

### 4.2 示例
```
feat(合同管理): 新增合同模板功能
fix(用户管理): 修复登录验证问题
docs: 更新API文档
```

## 5. 变更日志规范

### 5.1 变更类型
- `新增`：新功能
- `修复`：bug修复
- `变更`：现有功能变更
- `弃用`：即将删除的功能
- `移除`：已删除的功能
- `安全`：安全相关更新

### 5.2 格式要求
- 使用中文描述变更内容
- 每个变更项以短横线开头
- 关联对应的Issue编号（如 #123）

## 6. 标签管理

### 6.1 标签命名
- 版本标签：`vX.Y.Z`
- 预发布标签：`vX.Y.Z-alpha.1`、`vX.Y.Z-beta.1`

### 6.2 标签说明
- 每个发布版本必须创建标签
- 标签必须包含版本说明
- 标签必须推送到远程仓库

## 7. 文件管理

### 7.1 版本相关文件
- `VERSION`：当前版本号
- `CHANGELOG.md`：变更历史记录
- `.gitignore`：Git忽略文件配置

### 7.2 文档要求
- 所有API变更必须更新文档
- 重大功能变更需要更新用户手册
- 数据库变更需要更新数据库设计文档

## 8. 紧急修复流程

1. 从 `main` 分支创建 `hotfix/*` 分支
2. 修复问题并测试
3. 合并到 `main` 和 `develop` 分支
4. 更新版本号和变更日志
5. 创建新的发布标签
